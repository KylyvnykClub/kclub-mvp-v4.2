# План: Home Page для product-core (12 секций) + locale-routing

## Решения по вашим ответам
- **Локализация**: полный locale-routing через `next-intl` + сегмент `[locale]` + middleware redirect. Требует **ADR-010** (механизм локализации — undecided по ADR-005/SEO.md). EN полностью; RU/UK — каталоги с той же структурой (базовый перевод маркетингового копирайта, чтобы закрыть key-parity).
- **Contact form**: front-end-only с Zod-валидацией, **без** сохранения/контракта/Prisma/route (это отдельный feature-slice). Схема живёт в `packages/validation` (boundary-слой). Явный комментарий в UI: «не персистит».
- **Stats/testimonials**: честные placeholders без выдуманных метрик (концептуальные обещания без цифр; testimonials как нейтральная quote-структура). Никакого JSON-LD для них (нет supporting data).
- **Визуальный слой**: расширение CSS — новый `components.css` (которого DESIGN_SYSTEM требует, но нет) + расширение `tokens.css`. React-примитивы в `@kclub/ui` НЕ строим (отдельная задача).

## Что меняется по слоям (сверху вниз)

### 1. ADR-010 (новый) — `docs/adr/ADR-010-app-router-i18n.md`
- Decision: `next-intl`, App Router сегмент `[locale]`, `localePrefix: 'always'` (каждая страница имеет префикс → `/` редиректит на `/en`), `defineRouting`, locales `['en','ru','uk']`, default `en`.
- Задаёт ownership сообщений: **локальные JSON-каталоги `messages/<locale>.json` в product-core на старте** (не Directus); миграция на Directus — отдельный будущий slice (ADR-005 говорит, что Directus владеет переводами, но механизм потребления не определён — этот ADR закрывает пробел на MVP-стадии).
- Фиксирует Next 16 middleware/proxy file convention (проверю при реализации).

### 2. `packages/ui` — расширение дизайн-системы
**`src/tokens.css`** (расширить, всё в `:root`, namespace `--kc-`):
- Типографическая шкала: `--kc-font-size-{xs,sm,md,lg,xl,2xl,3xl}` в rem; `--kc-line-height-{tight,snug,normal,relaxed}`; `--kc-font-weight-{regular,medium,semibold,bold}`; `--kc-font-display` (для заголовков).
- Radius: `--kc-radius-{sm,md,lg,full}`.
- Shadow: `--kc-shadow-{sm,md,lg}`.
- Z-index: `--kc-z-{base,sticky,header,overlay}`.
- Spacing (расширить): `--kc-space-{1,2,3,6,10,12,20,24,32}`.
- Семантические: `--kc-color-surface`, `--kc-color-surface-muted`, `--kc-color-accent-strong` (hover), `--kc-color-border`.
- Border-width, transition/duration/easing.

**`src/components.css`** (новый; DESIGN_SYSTEM описывает его, но файла нет):
- `kc-section` (вертикальный ритм секций), `kc-cluster` (inline-кластер), `kc-card`, `kc-grid` (responsive grid), `kc-button` с `data-tone` (primary/neutral) и `data-size`, `kc-link`, `kc-nav`, `kc-badge`, `kc-form-field`, `kc-input`, `kc-textarea`, `kc-label`, `kc-footer`.
- Импорт-цепочка: `styles.css` → `tokens.css` + `components.css` (единый entry `@kclub/ui/styles.css` сохраняется, exports не меняются).

### 3. `packages/validation` — boundary-схема contact-формы
- **`src/contact.ts`** (новый): `contactFormSchema` (Zod) — `name` (nonempty), `email` (email), `message` (min len), опционально `company`/`phone`. Без side-effects (правило validation-слоя).
- **`src/index.ts`**: реэкспорт `contactFormSchema`.
- **`tests/contact.test.ts`** (новый): valid/invalid кейсы для покрытия 90% (coverage-порог в `vitest.config.ts`).

### 4. `apps/product-core` — i18n + структура App Router
- **Установить `next-intl`** (dependency product-core).
- **`src/i18n/routing.ts`**: `defineRouting({ locales, localePrefix: 'always', defaultLocale: 'en' })`.
- **`src/i18n/navigation.ts`**: `createNavigation(routing)`.
- **`src/i18n/request.ts`**: `getRequestConfig` — загрузка `messages/<locale>.json`, валидация locale.
- **`src/middleware.ts`** (или `proxy.ts` для Next 16 — проверю): `createMiddleware(routing)`, matcher исключает api/_next/static.
- **`messages/en.json`**: полный каталог (nav, 12 секций, footer, форма, ошибки валидации).
- **`messages/ru.json`**, **`messages/uk.json`**: та же структура, базовый перевод.
- **Реструктуризация App Router**:
  - Удалить `src/app/layout.tsx` + `src/app/page.tsx`.
  - **`src/app/[locale]/layout.tsx`** (новый root layout): `NextIntlClientProvider`, `generateStaticParams(locales)`, `<html lang={locale}>`, `generateMetadata` (title template, description, `metadataBase`, OG, `alternates.languages` hreflang en/ru/uk/x-default — требование SEO.md).
  - **`src/app/[locale]/page.tsx`**: compose страницы из секций.
- **Feature-папка** (по feature-module pattern): `src/features/home/components/` — 12 серверных компонентов-секций, каждый с `data-section="..."` (конвенция из существующего `page.tsx`):
  `Header, Hero, Stats, About, Plan, HowItWorks, Steps, Testimonials, Cta, Faq, ContactForm, Footer`.
- **Клиентские компоненты** (`'use client'`, минимальные, используют `kc-` классы — без ввода новых примитивов): `MobileNav` (бургер), `FaqAccordion`, `ContactForm` (клиентская Zod-валидация через `contactFormSchema`, success-состояние, **fetch/route отсутствует** — явно в комментарии).

## Честные ограничения (фиксирую явно)
- Contact form **не сохраняет** данные — бэкенд-feature-slice (contracts + Prisma + command + route + audit) отдельной задачей.
- Stats/testimonials — без метрик и без выдуманных данных (SEO.md: "no invented marketing data"). Заполнятся при появлении реальных данных.
- next-intl локально; миграция переводов в Directus CMS — будущий slice (ADR-005 ownership сохранён, механизм потребления зафиксирован в ADR-010 как временный).
- React-примитивы в `@kclub/ui` не строятся — только CSS-слой.
